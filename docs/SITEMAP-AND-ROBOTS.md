# How the sitemap and robots.txt are created

This document describes how `public/sitemap.xml` and `public/robots.txt` are produced, where the URL list comes from, and how the project keeps them in sync with the app.

**Site:** https://www.disputeforensic.com

## Overview

Both files are generated by a Node script rather than maintained by hand. That keeps every public URL aligned with the same inventory the verification tooling uses.

| Output | Path | Generator |
|--------|------|-----------|
| XML sitemap | `public/sitemap.xml` | `scripts/generate-seo.ts` |
| robots.txt | `public/robots.txt` | `scripts/generate-seo.ts` |

Next.js `src/app/sitemap.ts` and `src/app/robots.ts` are **not used** — static files in `public/` are served at `/sitemap.xml` and `/robots.txt`.

## URL inventory (`buildPublicUrlInventory`)

The script imports `buildPublicUrlInventory` from `src/lib/seo/publicUrlInventory.ts`. That function assembles all paths that should appear in search, then maps them to absolute URLs with the canonical host `https://www.disputeforensic.com` (or `NEXT_PUBLIC_SITE_URL`).

The inventory merges:

- **Static app routes** — A curated list (`APP_STATIC_PATHS`) of first-class marketing pages (home, services, forensic disciplines pillar, hubs, fees, FAQ, glossary, etc.).
- **Discipline pages** — Slugs from `src/data/disciplines.ts` as `/disciplines/:slug` (×4).
- **Case type pages** — Slugs from `src/data/case-types.ts` as `/case-types/:slug` (×10).
- **Sector pages** — Slugs from `src/data/sectors.ts` as `/sectors/:slug` (×6).
- **Guide pages** — Slugs from `src/data/guides.ts` as `/guides/:slug` (×6).

The combined list is deduplicated, sorted, and exposed as `allUrls` (absolute) and `allPaths` (pathname only).

### Excluded from sitemap

| Path | Reason |
|------|--------|
| `/contact` | Indexable in metadata; excluded from sitemap only |
| `/thank-you` | `noindex, nofollow` |
| `/privacy` | `noindex, follow` |
| `/terms` | `noindex, follow` |

## Sitemap generation (`scripts/generate-seo.ts`)

For each URL in `inventory.allUrls`, the script emits a standard sitemaps.org `<url>` entry with:

- `<loc>` — The absolute URL.
- `<lastmod>` — The calendar date when generation runs (YYYY-MM-DD in UTC).
- `<changefreq>` — Heuristic by path (home and pillar/guides hubs: `weekly`; others: `monthly`).
- `<priority>` — Heuristic by path (see `docs/SEO-ARCHITECTURE.md` Appendix B).

The full document is written to `public/sitemap.xml`. The `public/` directory is created if needed.

## robots.txt generation (`scripts/generate-seo.ts`)

`robots.txt` is a template in the same script (`renderRobots()`), driven by `ROBOTS_DISALLOW_PATHS` in `publicUrlInventory.ts`. It:

- Allows all crawlers on `/` by default.
- Disallows paths that should not be crawled as normal HTML: `/thank-you`, `/privacy`, `/terms`, `/api/`, `/_next/`, `/.netlify/`.
- Declares the sitemap location: `Sitemap: https://www.disputeforensic.com/sitemap.xml`.

The file is written to `public/robots.txt`.

## Commands

From the repo root:

```bash
npm run seo:generate   # Regenerate sitemap.xml + robots.txt
npm run seo:verify       # Fail if sitemap/robots drift from inventory
npm run seo:verify:ssr   # Placeholder for future head-tag audits
```

`npm run build` runs `seo:generate` before `next build`.

Run `seo:generate` after:

- Adding static routes to `APP_STATIC_PATHS`
- Adding new discipline, case-type, sector, or guide slugs in data files
- Changing robots disallow rules
- Before a release when you want fresh `lastmod` dates

## Continuous integration

`.github/workflows/seo-checks.yml` runs on pull requests and manual dispatch:

1. `npm ci`
2. `npm run seo:generate` — Ensures generation succeeds.
3. `npm run seo:verify` — Compares `public/sitemap.xml` and `robots.txt` to `buildPublicUrlInventory()`.
4. `npm run seo:verify:ssr` — Placeholder (passes until head audits are added).
5. Fails if committed `public/sitemap.xml` or `public/robots.txt` differ from freshly generated output.

## Deployment (Netlify)

`netlify.toml` sets explicit content types for crawlers:

- `/sitemap.xml` → `Content-Type: application/xml`
- `/robots.txt` → `Content-Type: text/plain`

Files in `public/` are served from the site root at `https://www.disputeforensic.com/sitemap.xml` and `https://www.disputeforensic.com/robots.txt`.

## Operational notes

- **Canonical host** is `https://www.disputeforensic.com` in `publicUrlInventory.ts` and the robots `Sitemap:` line. It must match `middleware.ts` (apex → www). Update both and regenerate if the primary domain changes.
- **New static pages** must be added to `APP_STATIC_PATHS` or `seo:verify` will fail after the next generate.
- **New dynamic pages** are picked up automatically when slugs are added to the relevant `src/data/*.ts` files.
- **Do not edit** `public/sitemap.xml` or `public/robots.txt` by hand.
